# AGENTS.md

Guidance for AI coding agents working in the PinConnect repository. Human-facing documentation lives in [`docs/`](docs/) and [`README.md`](README.md); this file covers how the pieces fit together and the conventions to follow when changing them.

## What this project is

PinConnect turns a photo of a PCB into an interactive pinout diagram. It is three loosely coupled tools that form a pipeline:

1. **pinout_design** — a static browser app that writes a board TOML config.
2. **pinout_gen** — a Python CLI that reads that TOML and renders self-contained interactive HTML.
3. **pinout_embed** — a Python-Markdown extension that embeds the generated HTML into MkDocs / Zensical sites.

The board TOML config is the interface between stage 1 and stage 2; the generated HTML is the interface between stage 2 and stage 3.

## Repository layout

- `pinout_gen/` — the `pinout-gen` CLI package.
  - `pinout_gen/pinout_gen/` — source: `cli.py` (argparse entry point), `config.py` (TOML dataclasses + loaders), `renderer.py` (SVG/HTML generation).
  - `pinout_gen/pinout_gen/connectors/` — connector type definitions (`*.toml`). **Source of truth** for connector geometry. Bundled with the package so they are found regardless of where the tool is run.
  - `pinout_gen/example.toml` — example board config.
- `pinout_design/` — the visual designer (vanilla ES modules, no framework).
  - `js/` — `board-model.js`, `state.js`, `*-panel.js`, `svg-renderer.js`, `toml-io.js`, `main.js`.
  - `connectors/` — JSON mirrors of the connector types, plus `index.json`. **Generated**, not hand-edited.
  - `lib/` — esbuild-bundled vendor JS (CodeMirror, smol-toml). **Generated.**
  - `tools/` — `convert-connectors.py`, `bundle-deps.mjs`, `package.json`.
- `pinout_embed/` — single-module Markdown extension (`pinout_embed.py`).
- `docs/` — human documentation.

## Key facts and gotchas

- **The designer must be served over HTTP.** It loads its connector library with `fetch()`, so opening `index.html` via `file://` fails. Use `python -m http.server` from `pinout_design/`.
- **Two representations of connector types, kept in sync.** `pinout_gen/pinout_gen/connectors/*.toml` is canonical. The designer uses JSON mirrors in `pinout_design/connectors/`, generated by `pinout_design/tools/convert-connectors.py` (which imports `pinout_gen.config`). If you change a connector TOML, regenerate the JSON — and update `pinout_design/connectors/index.json` if you add or remove a type.
- **Adding a connector `style` requires code, not config.** Styles (`box`, `latch`, `grid`, `xt30`) are drawn by custom code. A new style must be implemented in **both** `pinout_gen/pinout_gen/renderer.py` and the designer's `pinout_design/js/svg-renderer.js`, or the two will disagree.
- **A new `style` must be drawn in detail and match the real connector.** Study the existing `xt30` and `grid` styles as the quality bar: the render should reproduce the actual connector's housing shape as closely as possible, not a generic rectangle. Critically, the polarizing/keying features (latches, chamfers, cavities, the flat/keyed side) must be clearly visible in the render so a human can read the connector's orientation at a glance and not miswire it — this is a safety property, not a cosmetic one. A front cross-section DXF of the connector is the ideal reference to model the geometry from.
- **By default the generated HTML references the board image by relative path** (it is not embedded). The `-i` / `--image-embed` flag base64-encodes the image into the HTML instead. Any change to output handling must preserve both modes.
- **Board config vs. connector type are different TOML schemas.** See `config.py` for the exact dataclasses (`Board`/`Connector`/`Pin` vs. `ConnectorType`/`ConnectorGeometry`). Unknown geometry keys are silently ignored on load.
- **`pinout_embed` depends on the `attr_list` Markdown extension** for its `{ type=application/pinout }` tag syntax.

## Common commands

Run the designer:

```bash
cd pinout_design && python -m http.server 8000   # open http://localhost:8000
```

Install and run the generator (Python 3.9+):

```bash
pip install -e ./pinout_gen        # editable install for development
pinout-gen pinout_gen/example.toml   # writes board.pinout.html next to the config
```

Regenerate the designer's connector JSON after editing a connector type:

```bash
python pinout_design/tools/convert-connectors.py
```

Rebuild the designer's bundled vendor libraries:

```bash
cd pinout_design/tools && npm install && node bundle-deps.mjs
```

Install the Markdown extension:

```bash
pip install ./pinout_embed
```

## Conventions

- **Docs use one line per paragraph.** GitHub's Markdown viewer soft-wraps, so do not hard-wrap prose in `docs/` or the READMEs — keep each paragraph on a single line.
- **Keep the docs in step with behavior.** The board TOML reference (`docs/pinout-gen/board-toml.md`), connector types (`docs/pinout-gen/connector-types.md`), and `config.py` describe the same schema; change them together.
- **Generated and ephemeral directories are git-ignored** via the root `.gitignore` plus a `*` `.gitignore` inside each throwaway directory (`venv/`, `build/`, `*.egg-info/`, `__pycache__/`, `.claude/`). Do not commit generated output (`*.pinout.html`, `pinout_design/lib/` bundles, `pinout_design/connectors/*.json` are build artifacts).
- **No new runtime dependencies without reason.** `pinout_gen` relies only on the standard library (`tomllib`, with a `tomli` fallback below 3.11); the designer is dependency-free vanilla JS at runtime. Keep it that way unless there is a strong case.

## Verifying changes

- After editing `renderer.py` or a connector type, regenerate `pinout_gen/example.toml` and open the HTML to eyeball the result — connectors do not need to sit on a real board image to be inspected.
- After editing a connector type, run `convert-connectors.py` and confirm the designer still lists and renders the type.
- There is no automated test suite; validate by generating output and viewing it.
